= Enterprise Architect QEA file format
:toc:
:toclevels: 3

== Purpose

This document describes the Enterprise Architect QEA file format and how LutaML parses and transforms it into UML models.

== General

=== What is QEA?

QEA (Quantified EA) is Enterprise Architect's native database format based on SQLite 3. It stores complete UML models including:

* Packages, classes, interfaces, enumerations, data types
* Attributes, operations, and parameters
* Associations, generalizations, and other relationships
* Diagrams with visual layout information
* EA-specific metadata (stereotypes, tagged values, constraints)
* Cross-references and extended properties

=== Why direct QEA parsing?

Direct QEA parsing offers significant advantages over XMI export:

Performance:: 10-20x faster than XMI parsing (1-2 seconds vs 20-40 seconds for a 20MB model)
No export step:: Parse directly from EA database without manual export
EA metadata:: Access to EA-specific information not available in standard XMI
Memory efficient:: Streams data from SQLite without loading entire file into memory

=== File structure

QEA files are SQLite 3 databases with a specific table schema defined by Enterprise Architect.

.QEA file characteristics
[source]
----
File Type: SQLite 3 database
Extension: .qea
Typical Size: 5-50MB for medium models
Tables: ~95 total tables (42 non-empty in typical models)
Encoding: UTF-8
----

== Architecture

=== Two-layer transformation pipeline

LutaML uses a strict two-layer architecture to transform QEA data into UML models:

[source]
----
┌────────────────────────────────────────────────────────────────┐
│ QEA DATABASE (SQLite)                                          │
│ - Flat relational tables (t_object, t_connector, etc.)        │
└────────────────────────────────────────────────────────────────┘
                            ↓
┌────────────────────────────────────────────────────────────────┐
│ LAYER 1: QEA PARSE LAYER (Lutaml::Qea::Models)               │
│ - EaObject, EaConnector, EaDiagram, etc.                      │
│ - Direct 1:1 mapping to database rows                         │
│ - Flat relational structure matching database schema          │
└────────────────────────────────────────────────────────────────┘
                            ↓
┌────────────────────────────────────────────────────────────────┐
│ TRANSFORMERS (Lutaml::Qea::Factory)                           │
│ - ClassTransformer, DiagramTransformer, etc.                  │
│ - Convert flat relational to tree-like structure              │
│ - Resolve references and build relationships                  │
└────────────────────────────────────────────────────────────────┘
                            ↓
┌────────────────────────────────────────────────────────────────┐
│ LAYER 2: UML DOMAIN LAYER (Lutaml::Uml)                      │
│ - Class (with nested Attribute[], Operation[], Constraint[]) │
│ - Package (with nested Package[], Class[])                    │
│ - Diagram (with nested DiagramObject[], DiagramLink[])        │
│ - Tree-like structure with rich domain behavior              │
└────────────────────────────────────────────────────────────────┘
----

=== Layer 1: QEA parse layer

==== General

The QEA parse layer provides direct representation of database tables as Ruby model objects.

Location:: [`lib/lutaml/qea/models/`](../lib/lutaml/qea/models/)
Purpose:: 1:1 mapping of SQLite rows to Ruby objects
Structure:: Flat relational data matching database schema
Base class:: [`Lutaml::Qea::Models::BaseModel`](../lib/lutaml/qea/models/base_model.rb)

==== Characteristics

* Each table maps to one model class
* Column names map directly to attributes
* No nested relationships (references stored as IDs)
* No business logic or domain behavior
* Serializable using `lutaml-model` gem

.Example QEA model class
[source,ruby]
----
class EaObject < BaseModel
  attribute :object_id, :integer
  attribute :object_type, :string
  attribute :name, :string
  attribute :package_id, :integer  # Foreign key, not resolved
  attribute :stereotype, :string
  # ... additional attributes matching database columns
end
----

=== Layer 2: UML domain layer

==== General

The UML domain layer provides rich, hierarchical UML model objects.

Location:: [`lib/lutaml/uml/`](../lib/lutaml/uml/)
Purpose:: Tree-like domain model with nested object hierarchies
Structure:: Objects contain child objects (not just IDs)
Namespace:: ALL UML elements MUST be in `Lutaml::Uml` namespace

==== Characteristics

* Hierarchical structure with parent-child relationships
* Resolved references (objects contain other objects, not IDs)
* Domain-specific behavior and methods
* Follows UML metamodel conventions
* Used by all LutaML consumers

.Example UML domain class
[source,ruby]
----
class Class
  attribute :name, :string
  attribute :attributes, Attribute, collection: true  # Nested objects
  attribute :operations, Operation, collection: true  # Nested objects
  attribute :constraints, Constraint, collection: true  # Nested objects
  attribute :generalizations, Generalization, collection: true
  # Rich domain behavior...
end
----

=== Transformation layer

==== General

Transformers convert QEA models to UML domain objects.

Location:: [`lib/lutaml/qea/factory/`](../lib/lutaml/qea/factory/)
Purpose:: Convert flat relational data to tree-like domain model
Base class:: [`Lutaml::Qea::Factory::BaseTransformer`](../lib/lutaml/qea/factory/base_transformer.rb)

==== Transformation responsibilities

Reference resolution:: Convert object IDs to actual object references
Hierarchy building:: Create parent-child relationships
Data mapping:: Map EA concepts to UML metamodel
Validation:: Ensure data integrity during transformation

.Example transformer
[source,ruby]
----
class ClassTransformer < BaseTransformer
  def transform(ea_object, database)
    uml_class = Lutaml::Uml::Class.new
    uml_class.name = ea_object.name

    # Resolve attributes (nested objects, not IDs)
    uml_class.attributes = load_attributes(ea_object.object_id, database)

    # Resolve operations
    uml_class.operations = load_operations(ea_object.object_id, database)

    # Resolve constraints
    uml_class.constraints = load_constraints(ea_object.object_id, database)

    uml_class
  end
end
----

== Configuration

=== Schema configuration file

QEA parsing is configured via [`config/qea_schema.yml`](../config/qea_schema.yml).

.Configuration structure
[source,yaml]
----
version: "1.0"
description: "EA QEA database schema configuration"

model_classes:
  t_object: Lutaml::Qea::Models::EaObject
  t_attribute: Lutaml::Qea::Models::EaAttribute
  # ... other tables

tables:
  - table_name: t_object
    enabled: true
    primary_key: Object_ID
    collection_name: objects
    description: "UML objects (classes, interfaces, etc.)"
    columns:
      - name: Object_ID
        type: INTEGER
        primary: true
      # ... other columns

transformations:
  object_types:
    Class: Lutaml::Uml::Class
    Interface: Lutaml::Uml::Class
    # ... other mappings

  transformers:
    class: Lutaml::Qea::Factory::ClassTransformer
    # ... other transformers
----

=== Boolean field handling

EA stores boolean values as INTEGER (0/1). The configuration specifies which fields should be treated as boolean:

.Boolean fields
[source,yaml]
----
boolean_fields:
  - IsStatic
  - IsCollection
  - IsOrdered
  - AllowDuplicates
  - IsControlled
  - Protected
  - IsRoot
  - IsLeaf
  - IsSpec
  - IsActive
  - Const
  - Tagged
----

== Usage

=== Parsing a QEA file

.Basic parsing
[source,ruby]
----
require "lutaml/qea"

# Parse QEA file directly
document = Lutaml::Qea.parse("model.qea")
# => Returns Lutaml::Uml::Document

# With options
document = Lutaml::Qea.parse("model.qea",
  include_diagrams: true,
  validate: true
)

=== Optimization techniques

The QEA parser uses several optimization techniques:

Streaming:: Reads data from SQLite without loading entire file
Lazy loading:: Loads only requested tables
Indexed queries:: Uses database indexes for fast lookups
Batch processing:: Processes related records together
Connection pooling:: Reuses database connections

== Limitations

=== File format requirements

* QEA files must be SQLite 3 format
* Requires EA-specific table structure
* Unicode (UTF-8) encoding required

=== Feature support

* Read-only access (does not write back to EA database)
* Some EA proprietary features may not be fully supported
* Diagram layout is approximate (EA uses proprietary rendering)
* Version control information not extracted

=== Compatibility

* Tested with EA versions 13-16
* May not support very old EA versions (< 10)
* Custom EA schema modifications not supported
